### Известные ограничения (A2/A3/A6) — с глубоким фиксом

**A2 — неатомарность `payDebt` (double-tap / два editor'а → переплата).** Задокументировано, но остаётся реальным дефектом целостности (находки :1419/:1425/:1427). Точечные патчи лечат симптомы. **Глубокий фикс:** обернуть весь `payDebt` в один `$transaction` с `SELECT ... FOR UPDATE` на liability-`FinAccount` (как `WalletBalance` под row-lock в кошельке). Внутри лока: пересчитать `remaining` → `amount = min(requested, remaining)` → записать проводку → пересчитать `left` из свежего баланса → выставить `debtClosedAt` только если `left<=0`. Это одновременно закрывает переплату (:1419), stale-close (:1427) и stale-`left` (:1425) — три находки одним изменением. Заодно ввести опциональный `paymentToken`/idempotency-key для защиты от двойного тапа на клиенте.

**A3 — неатомарность/гонки `checkBudgetThresholds` (кросс-счёт, пропуск/дубль пересечения).** Находки :953 и :938. **Глубокий фикс:** передавать в `checkBudgetThresholds` **pre-transaction snapshot** суммы категории, вычисленный внутри той же `$transaction`, что пишет проводку (а не пересобирать `spentBefore` вычитанием из глобального SQL-суммарного после коммита). Дедуп срабатываний — по ключу `(bookId, categoryId, period, threshold)` в отдельной таблице/Redis-множестве, чтобы два конкурентных пересечения не слали два пуша. Период — из **того же** источника, что и отчёт (устраняет :938). Порог 100 000 при 9000+1000+1000 тогда детектится ровно один раз.

**A6 — TZ-класс (весь модуль считает дни/месяцы в UTC).** Находки :1873 (календарь), :1676 (дедлайн-напоминания), :906 (тренд), плюс потенциальный :51 (fromDbDate). Для рынка KZ (UTC+5) это систематический сдвиг на границе суток/месяца. **Глубокий фикс:** ввести единый helper-слой времени с TZ пользователя (в v1 можно захардкодить `Asia/Almaty`, поле `user.timezone` уже упоминается в профиле). Все `getUTCDate/getUTCMonth/getUTCDay` в finance заменить на «локальный день/месяц в TZ книги»; дедуп-окно напоминаний — локальный день; `getTrend` и период отчёта — из одного `localToday()`-источника. Один слой закрывает три находки и снимает будущие TZ-баги при вводе mobile.

**Вывод по разделу 1:** денежное ядро крепкое — нет IDOR на запись, нет минусов на кошельке, двойная запись цела. Основная масса дефектов — это (а) **пропуск бюджет-чека на update-путях**, (б) **archived-счета не отклоняются в create-путях**, (в) **TZ-в-UTC**, (г) **неатомарность payDebt/threshold под конкуренцией общей книги**. Четыре системных фикса (budget-check на всех путях, archived-гарда в loadPair, TZ-слой, payDebt в одной транзакции) закрывают ~15 из ~25 находок.

## 4. Архитектурные заметки (перенять из OSS)

- **Rules-движок — декларативный, не код.** Firefly держит `triggers[] + actions[]` в JSON и **сознательно простым** (без regex/вложенной логики — «сложные rule-движки никто не использует»). У нас уже есть реестровый паттерн (quick-actions, rich-cards, process-node-registry) — правила финансов = реестр операторов-условий + реестр действий, тем же способом. Бенчмарк: не переусложнять.
- **Импорт-профиль как переиспользуемый JSON.** Firefly Data Importer: конфиг `{колонка→роль, mapping значений}` скачивается и прогоняется повторно. Для KZ — фундамент под Kaspi Gold PDF/Halyk: настроил маппинг один раз → сохранил профиль → импорт в клик. Парсер — **детерминированный под каждый банк** (pdfplumber), не «универсальный AI» (подтверждено bank2excel/IziBiz: рукописный парсер надёжнее; главная боль — склейка строк и многострочные описания в Kaspi-PDF).
- **Payee как first-class сущность.** Отдельно от нашего `personUserId` (близкий человек). Мерчант (Magnum/Netflix) — своя таблица `FinPayee` с авто-матчингом по строке выписки; отчёт «сколько в Magnum». Замыкает пару с импортом и rules.
- **Rollover-семантика конвертов (Actual).** Остаток/перерасход категории переносится между месяцами; «доступно» = лимит + перенос − факт. Наш `FinBudget` помесячно-независим — добавить поле «переносить остаток» + пересчёт.
- **Net worth = агрегат поверх типизированных счетов.** Активы (asset) − пассивы (liability) во времени. Данные уже есть, нужен только агрегат+график. Maybe подтверждает наш `FinAccount.kind`-подход (delegated types вместо наследования) — расширяемо на инвестиции без раздувания схемы.
- **Материализация баланса (истина в журнале, кэш-баланс под row-lock).** Приём уже применён в нашем wallet-леджере (`WalletBalance`); при росте объёма finance-книги баланс счёта стоит материализовать так же — заодно снимает Number-precision-риск display-DTO (:198).
- **AI function-calling = набор узких read-инструментов** (`GetTransactions/GetAccounts/GetBalanceSheet/GetIncomeStatement`, Maybe/Sure) — ровно наш Принцип 4 (чтение через `core/access`, действия через API). Каждый finance-эндпоинт уже = готовый инструмент.

## 5. Что бы я сделал первым (максимум ценности / минимум усилий)

1. **Budget-check на всех write-путях + один источник периода.** Вызвать `checkBudgetThresholds` в `updateTransaction` и считать период из того же источника, что и отчёт (закрывает 🔴:565/:594 и 🔴:938 — тихий пропуск алерта 80/100% при правке и off-by-one дате). Это ядро продуктового обещания «план-факт», сейчас дырявое. **Усилие S, ценность 5.**

2. **Archived-гарда в `loadPair` (+ upsertBudget/createDebt/recordOperationForBook).** Один `archived:false` закрывает 🔴:497, 🟠:1939, 🟢-recurring-archived — «оживающие» скрытые счета и запись процессов в закрытый счёт. **Усилие S, ценность 4.**

3. **Главное число «Свободно до конца месяца» на первом экране.** Агрегат из уже имеющихся данных над гридом (доход − факт − остаток лимитов − предстоящие долги/повторы). Отвечает на вопрос №1 fintech, снимает главную UX-претензию (нет ответа «сколько можно тратить»). **Усилие S, ценность 5.**

4. **`payDebt` в одну транзакцию с FOR UPDATE (глубокий фикс A2).** Читать остаток + писать платёж + решать close внутри одного лока на liability-счёт (как `WalletBalance` в кошельке) — тремя строками логики закрывает переплату (:1419), stale-close (:1427) и stale-`left` (:1425). Целостность денег важнее косметики. **Усилие S-M, ценность 4.**

5. **Снять хардкод `'KZT'` + мобильный приоритет ввода.** (а) Валюта из счёта/правила в долгах/повторах/превью (🟠 F5 — неверные числа = потеря доверия при мультивалюте); (б) на мобиле форма ввода первой (`order:-1` / FAB) — ежедневный сценарий сейчас утоплен под 5 панелей. Оба — малое усилие, прямой удар по удержанию. **Усилие S-M, ценность 4.**

Эти пять закрывают самый острый денежный дефект (budget-check), самый широкий класс (archived), главную UX-претензию (нет главного числа), самый рискованный по целостности путь (payDebt) и самый заметный визуальный баг (валюта/мобайл) — при минимальном коде. Следом — TZ-слой (глубокий A6, разблокирует mobile) и rules-движок+импорт Kaspi (стратегический killer для KZ).
