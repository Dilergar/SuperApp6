## Problem
Sharing/visibility/access is re-implemented per module with divergent semantics:
- Card visibility: UNION of per-Circle JSONB flags (`mergeVisibilities`), `resolveVisibilityForViewer` (contacts.service.ts).
- Calendar: MAX of none<busy<detailed via `CalendarShare` + `Circle.calendarVisibility`, `resolveAccessLevel` (calendar.service.ts) ‚Äî called per included calendar = N+1 (√ó2 queries each).
- Showcase: `ShowcaseShare(principalType user|circle)`, `canViewShowcase`/`ownerCirclesViewerIsIn`/`circlesViewerBelongsTo` (shop.service.ts) ‚Äî last one full-scans all viewer ContactLinks.
- Task group assign: snapshot into TaskParticipant. Workspace profile: role-based field masking. Wallet holders: hardcoded issuer-only.
The "find owner's Circles containing viewer via ContactLink‚ÜíCircleMembership" walk is COPIED 3√ó with different merge logic. No reverse indexes; no access cache; in-JS JSONB merge.

## Locked decisions
1. **Full ReBAC** (relation tuples + rewrite rules), not a flat grant table ‚Äî inheritance (org‚Üíemployee cards, shop‚Üíshowcase, editor‚áíviewer) must be a RULE, not per-service code.
2. **Field-level visibility = thin layer ON TOP** (engine answers "who sees record / which Groups is viewer in"; field flags age/email applied above, as today). Salesforce FLS-vs-record split.
3. **Source of truth = PROJECTION.** One uniform tuple store. Domain tables (Circle/CircleMembership, UserRole, participants, future org structure) stay authoritative for data + UI; their ACCESS-RELEVANT EDGES are projected into tuples & kept fresh: synchronous dual-write inside existing Prisma transactions + EventBus `AuthzProjectionListener` + periodic reconciler (same discipline as wallet ledger). Pure-sharing tables `CalendarShare`/`ShowcaseShare` MIGRATE fully into tuples & drop. NOT federation/adapters (reverse "what can I see" queries are the product's core ‚Üí uniform store wins). Don't import whole DB ‚Äî only access edges; transitive closure derived at query time. Runtime facts (active workspace, time) ‚Üí contextual, not stored.
4. **Build own minimal engine in-monolith** (`core/access`) on existing Postgres, data shape SpiceDB/OpenFGA-compatible so later migration to a dedicated engine needs no model change. No separate SpiceDB service now (ops overkill, solo team).
5. **Capability-based checks**: code checks capability keys (`task.accept`, `showcase.view`), NOT role names. Permission keys = stable code vocabulary; new services add capabilities/resource-types/relations as CONFIG/DATA, no DB migration (OpenFGA immutable/modular-models property).

## Engine shape (`core/access`)
- Table `RelationTuple(resourceType, resourceId, relation, subjectType, subjectId, subjectRelation?)` + forward & reverse indexes. subjectRelation enables usersets (`circle:X#member`).
- Per-resource-type CONFIG (TS, versioned in code): relations, level ordering (calendar none<busy<detailed), rewrite rules (computed_userset: editor‚áíviewer; tuple_to_userset/arrow: showcase.manager ‚Üê shop.manager, card.viewer ‚Üê workspace policy).
- Resolver: `check(subject, capability, resource)` / `resolveLevel(...)` / `listObjects(subject, relation, type)` [reverse ‚Äî "what can I see"] / `listSubjects(...)`.
- Cache: Redis, keyed by (subject,resource)/(subject,owner-scope), epoch-invalidated; bump epoch on writes.
- Capability registry: code map of permission keys per service.
- NOT in the workspace chokepoint (access is cross-cutting, like wallet).
