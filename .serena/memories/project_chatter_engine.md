## Решения грилла (закрыты с пользователем — НЕ пересматривать)
1. v1-потребители: **Задачи** + **B2B-аудит** (найм/роли/должности) со страницей **«Журнал организации»** (доступ **manager+**, выбор пользователя).
2. **Системные плашки чатов = проекция хроники** («плашки = Chatter, читать удобно прямо в чате»). `TaskSystemListener` УДАЛЁН — плашки задач производит движок, 5 легаси-текстов байт-в-байт.
3. Followers — v2 (отдельная будущая таблица ChatterFollower, на модели ничего не заложено — не нужно).
4. Секции «История» на детальке задачи НЕТ — чат и есть лента (решение пользователя).

## Архитектура
- **Модель `ChatterEntry`** (`chatter_entries`, миграция `chatter_engine`): refType+refId (полиморфно, паттерн FileLink), FK-free (паттерн FinAuditLog), `id BigInt autoincrement` = append-log + курсор (паттерн LedgerTransfer; **в DTO всегда `.toString()` — JSON.stringify на BigInt бросает**), actorId? + `actorName` снапшот (переживает удаление аккаунта; null = система), typeKey (= ключ CHATTER_REGISTRY), `changes` JSONB `[{field,label,from,to}]` — display-ready строки (даты — детерминированно в `APP_TIMEZONE`, не TZ сервера), payload (переменные шаблона: targetUserId/targetName/roleLabel/positionName/**branchName raw**/taskTitle), `needsChatPost`+`chatPostedAt` (проекция джобами core/jobs; терминал = ремень идемпотентности обработчика). Индексы: (refType,refId,id) / (workspaceId,id) / (workspaceId,typeKey,id) / **частичный** `chat_post_pending (id) WHERE needs_chat_post AND chat_posted_at IS NULL` (пересоздан в `jobs_engine`, кормит бэкфилл). workspaceId денормализован. НЕ в chokepoint (`WORKSPACE_SCOPED_MODELS` опт-иновый). Хранится вечно (retention-крона нет осознанно).
- **Запись СИНХРОННО в транзакции мутации** — `ChatterService.log(tx|null, entry)` / `logMany`; вне tx = best-effort catch. Обоснование: шина at-most-once + события НЕ несут old-значений (проверено сканом всех emit'ов) → проектор с шины невозможен для диффов. `diffTracked(spec, before, after)` — чистый дифф-хелпер, spec = `Record<field, {typeKey, label, format(row)}>`, сравнение по display-строкам.
- **Доступ** = `ChatterRefRegistry.register(refType, {canView})` (паттерн FilesRefRegistry): 'task' = access.can(task.view) ∥ создатель ∥ участник (регистрирует TasksService); 'workspace' = роль ≥ manager (регистрирует WorkspacesService — HR-события не текут рядовым). Нового типа в access-schema НЕТ, tuple-проекции НЕТ.
- **Чат-проекция** (текущее состояние — на движке джобов): `CHATTER_REGISTRY` (shared, паттерн NOTIFICATION_REGISTRY) несёт `{template, icon, category: tasks|staff, chatPost}`; `renderChatterText(typeKey, entry)` (поверх общего `interpolateTemplate`) в shared — ЕДИНЫЙ рендер API-плашки и веб-журнала. `logMany` создаёт записи `createManyAndReturn` и для needsChatPost-строк ставит джоб `chatter.chatpost` (`payload {entryId}`, uniqueKey `ce:<id>`, maxAttempts 8) В ТОЙ ЖЕ транзакции. Обработчик (`ChatterService.handleChatPostJob`, регистрация в onModuleInit): row отсутствует/`chatPostedAt`/`!needsChatPost` → no-op; нет синка → throw (бэкофф → dead-letter в логах — синки регистрируются в onModuleInit потребителей, до старта воркера); `sink.post(dto)` → терминал `chatPostedAt` (guard null). Бэкфилл (onApplicationBootstrap): незапощенные записи окна 24ч БЕЗ джоба (проверка jobs по uniqueKey; discarded = решение «не переигрывать», не потеря) → enqueue. Синк регистрирует МЕССЕНДЖЕР (`ChatterChatSink.registerChatSink('task')` — направление CallsRecordingRegistry): `postTaskSystemMessage(refId, typeKey, renderChatterText(...), {chatterEntryId})`; **дедуп** — `postSystemMessage` findFirst по `payload.chatterEntryId` (повторный пост после краха «после поста, до терминала» не дублит) + `memberIds` ДО транзакции сообщения; eventType плашки = typeKey; веб читает только payload.text.
- **Чтение**: `GET /chatter/:refType/:refId?cursor&limit` (тонкий контроллер движка) → `{items, nextCursor, actors}` (actors = батч USER_LITE незанятых deletedAt для PersonChip; удалённый → фолбэк actorName). `GET /workspaces/:id/journal?category=` — `journal.controller.ts` в workspaces (path-based, паттерн staff/office), гейт ВНУТРИ `ChatterService.listJournal` через **зарегистрированный canView-резолвер 'workspace'** (движок RolesService/ранг-логику НЕ держит). GET-схемы НЕ `.strict()` (лишний query-параметр молча отбрасывается). Keyset `id < BigInt(cursor)` DESC, take limit+1. Индекс `(workspace_id, type_key, id)` под фильтр category.

Легаси-паритет: assigned/submitted/accepted/returned/completed байт-в-байт из удалённого TaskSystemListener, фолбэк «Кто-то».

CRUD справочников/профиль орг — волна 2 (низкая аудит-ценность).

## Gotchas
- CSS-классов `.surface-low`/`.body-md` в globals.css НЕТ — только display/title/label-*, card, ghost-border, wash-*; фоны инлайном var(--surface-container-lowest).
- `chatPostDefault` из CHATTER_REGISTRY: неизвестный typeKey → false (запись есть, плашки нет — safe).
- Хроника workspace-задач получает workspaceId из строки задачи (chokepoint инжектит его в tx.task.create при активном X-Workspace-Id).
- Плашка «Задача выполнена» у САМО-задачи создаёт её чат (паритет со старым слушателем — так было и раньше).
- verify-chatter (секции 12–13): инжектит ChatterEntry прямо через Prisma (мимо log() = без джоба) + вставляет джоб-строку руками (ровно то, что делает бэкфилл) → обработчик допощивает ≤15с; дедуп-тест: сброс chatPostedAt + НОВЫЙ джоб того же ключа (старый терминален — partial unique пускает) → плашка не дублится.
