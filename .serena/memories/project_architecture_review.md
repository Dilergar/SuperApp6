## Block 1 ‚Äî Money (do first)
- **CRITICAL double-settle** ‚Äî `apps/api/src/modules/wallet/ledger.service.ts:307-347`: `postPending`/`voidPending` check `isResolved` BEFORE `lock()`, no re-check after, no partial-unique on `pending_id WHERE kind IN ('post_pending','void_pending')` ‚Üí concurrent double "–ü—Ä–∏–Ω—è—Ç—å"/confirm = double payout. Fix: re-check after lock + partial unique index + status-guarded `updateMany` in `acceptWork` (`tasks.service.ts:618-628`) and `confirmOrder` (`shop.service.ts:690-722`).
- **HIGH order settlement rides at-most-once EventBus** ‚Äî `shop.events.ts:20-33` (`task.completed` ‚Üí `onFulfillmentDone` capture): lost event = buyer money frozen forever, no sweep. Fix: call `onFulfillmentDone` synchronously from task acceptance (it's idempotent), keep listener as safety net, add ShopCron sweep (confirmed + task done ‚Üí capture).
- **HIGH `withLock` no owner token/renewal** ‚Äî `shared/redis/redis.service.ts:67-94`: release = unconditional DEL (can delete another instance's lock); `fireDue` (`scheduled-message.service.ts:134-179`) doesn't claim rows ‚Üí duplicate scheduled sends on 60s TTL. Fix: uuid lock value + Lua compare-and-del; `updateMany pending‚Üísending` claim before sending.
- MEDIUM races: crowdfunding `withdraw` no FOR UPDATE on campaign (`shop.service.ts:594-613`, contribute HAS it at :546); `confirmOrder` non-atomic ‚Üí duplicate fulfilment task; resource booking capacity check outside tx (`calendar.service.ts:249-270`); P2002-recovery inside an open tx is dead in Postgres (aborted 25P02) ‚Äî `ledger.service.ts:85-94`, `escrow.service.ts:54-65`.

## Block 3 ‚Äî core/access perf wall (first thing to fall at ~thousands of users)
- Global epoch `acl:epoch` (`access.service.ts:95-151`): every grant/revoke bumps ‚Üí entire check-cache invalidated platform-wide; task create = 2 bumps (`resyncTaskRoles` = revoke-then-grant, `access-projection.service.ts:264-280`). Effective cache hit-rate ‚Üí 0 under live traffic; every `chat.view` check (per message/typing) = 2 Redis RTT + SQL graph walk.
- `listObjects` BFS = 1 SQL per visited node, no type pruning (`access-resolver.ts:107-134`); power user with hundreds of chats/tasks ‚Üí hundreds-thousands of SQL per call. Hot callers: calendar overlay √ó2 (`calendar.service.ts:163-168`), `/shop/accessible`, `/shop/orders/incoming`, wishlists, presence.
- Fixes: epoch per-resourceType; prune frontier by `ACCESS_SCHEMA` type-reachability (calendar ‚Üê only user/circle); batch frontier with one IN query; `resyncTaskRoles` ‚Üí `applyDiff` (order/event already do, `:291-296`); bump epoch AFTER COMMIT (currently inside tx ‚Üí up to 10-min stale-ACL windows, security-adjacent).

## Block 4 ‚Äî hot-path perf
messenger `listChats`: unread = groupBy over ENTIRE history (`messenger.service.ts:1045-1071`; should be `seq > lastReadSeq`) + includes all members of all chats, no pagination (:974-989). Calendar recurring masters fetched with no lower time bound forever (`calendar.service.ts:120-134`; also :702-710, :743-763) ‚Üí materialize `recurrenceEndsAt`. `/tasks`: offset+count pagination AND **visible bug** ‚Äî `orderBy priority` on string ‚Üí high sorts BELOW low (`tasks.service.ts:283-292`). Duplicate indexes: `messages` `@@unique`+`@@index` both on (chatId,seq) + 3 more (schema:1153-1154, 798-799, 232-236, 117-118). WS `read`/`delivered` accept arbitrary seq (no clamp to `chat.lastSeq`), no per-socket rate limit on typing/heartbeat. Reconcile crons are full-table scans (batch by cursor; drop nightly backfills ‚Äî they're one-time scripts).

## Block 5 ‚Äî session/realtime reliability
No single-flight refresh in web (`apps/web/src/lib/api.ts:28-48`) NOR mobile ‚Üí parallel 401s race-rotate the refresh token ‚Üí random logouts (daily on mobile cold start). Socket reconnects with stale access token (`useMessengerSocket.ts:71` ‚Äî token captured once) + no chat invalidation on connect (missed messages not caught up). logout-all doesn't disconnect live sockets (`messenger.gateway.ts:53-70` ‚Äî auth on handshake only).
